# TEC.ERP Serious Learning Simulation Principles

## Product nature

TEC.ERP is a **professional learning simulation** inside a Learning Operating System.

| TEC.ERP is | TEC.ERP is not |
|------------|----------------|
| Professional role assumption | A game |
| Authentic problem + evidence work | An LMS content shelf |
| Consequence-bearing enterprise Twin | A simple ERP sandbox |
| Facilitated classroom simulation | A text-heavy online course |
| Judgment under incomplete information | A chatbot experience |

## Twenty binding principles

1. **The learner assumes a professional role.** Screens speak as mandat / dossier / recommandation, not “exercise.”  
2. **Every mandate contains an authentic problem.** No mandate exists only to demonstrate a menu.  
3. **Important information may be incomplete.** Missing data is pedagogical, not a defect.  
4. **Investigation precedes action.** Hypothesis and evidence map before irreversible ERP writes (Simulation/Evaluation).  
5. **Decisions create consequences.** Important actions change Twin state, documents, KPIs, stakeholders, or risk.  
6. **Consequences affect multiple business dimensions.** Local wins may harm cash, service, compliance, or workforce.  
7. **Errors create learning opportunities.** Safe failure with recovery paths; not silent punishment.  
8. **Simulation requires facilitation.** Professor packages BEFORE/DURING/AFTER are mandatory.  
9. **Simulation requires debrief.** No major simulation closes without structured debrief.  
10. **Reflection requires reapplication.** Extracted principle must be tried in a short transfer task.  
11. **Guidance decreases with mastery.** Novice → intermediate → advanced; Professor may override.  
12. **The Digital Twin remains coherent.** Narrative never contradicts authoritative Twin facts.  
13. **Stakeholders react consistently.** Authority, incentives, and prior communication constrain ambient AI.  
14. **AI supports but does not replace judgment.** Visible AI coaches; ambient AI animates; neither decides for the learner.  
15. **Results are not reduced to points.** Evidence quality, process integrity, and defense matter more than scores.  
16. **Trade-offs may exist without one perfect answer.** Multiple defensible paths allowed when contract says so.  
17. **Evidence quality matters.** Weak, fabricated, or untraceable evidence fails professional standards.  
18. **Professional communication matters.** Executive synthesis, escalation, and stakeholder tone are assessable.  
19. **The Professor remains central.** Platform supports teaching; it does not replace facilitation.  
20. **Every simulation must have a learning purpose.** Realism without learning purpose is prohibited.

## Construct definitions

| Construct | Definition | Contract rule |
|-----------|------------|---------------|
| Functional realism | Fidelity needed for authentic decisions (process, controls, documents, KPIs, stakeholders) | Required |
| Unnecessary realism | Decorative complexity that adds load without learning value | Prohibited |
| Safe failure | Learner can err without permanent harm to historical runs or unrelated cohorts | Required; V1 immutable |
| Authentic pressure | Time, urgency, competing priorities that mirror professional work | Allowed; must be explainable |
| Ambiguity | Incomplete or conflicting evidence by design | Allowed by level; never random chaos |
| Consequence visibility | Learner and Professor can see what changed and why | Required for important actions |
| Continuity | Same enterprise Twin across modules/missions in a Run | Required for V2 simulation state |
| Replay and comparison | Revisit a decision path; compare alternatives/groups | Professor capability (Wave 7+) |
| Professor intervention | Freeze, reveal, trigger, suppress, compare, hide keys | Required capability contract |
